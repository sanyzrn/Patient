## ۰. تفاوت کلیدی با پلن قبلی (چرا این نسخه؟)

پلن قبلی ساختار خوبی داشت، اما با کد فعلی هم‌خوان نیست. سه مشکل داشت:

**الف) خیلی از کارهای «بحرانی» آن قبلاً انجام شده‌اند.** مخصوصاً در افزونه:

| موردی که پلن قبلی «باید انجام شود» گفته بود | وضعیت واقعی در کد |
|---|---|
| اضافه کردن `check_ajax_referer()` به endpointها | ✅ انجام شده — هر ۶ endpoint nonce دارند |
| encrypt کردن Gemini API key | ✅ انجام شده — AES-256-CBC با `AUTH_KEY` (نه فقط base64) |
| Capability check در پنل ادمین | ✅ انجام شده — `manage_options` روی همه صفحات و export |
| تنظیم timeout برای Gemini | ✅ انجام شده — ۶۰ ثانیه، قابل تنظیم با filter |
| BOM در خروجی CSV | ✅ انجام شده — `\xEF\xBB\xBF` prepend می‌شود |
| lazy load کردن assets چت‌بات | ✅ انجام شده — register/enqueue تفکیک شده |
| rate limiting قوی‌تر | ✅ انجام شده — مبتنی بر transient (IP+session)، نه فایل |
| **RAG (پلن قبلی: «فاز ۴، بالاترین ارزش»)** | ✅ **قبلاً ساخته شده** — جدول `kb` با FULLTEXT + chunking + hybrid retrieval |
| داشبورد آمار هوشمند | ✅ انجام شده — جدول stats اتمیک، CSAT، روند ۱۴ روزه |

**ب) دو مشکل بحرانیِ واقعی را کاملاً از قلم انداخته بود** (در ادامه — بخش ۱).

**ج) چند باگ وبسایت که «بحرانی» علامت خورده بودند، قبلاً رفع شده‌اند:**
- باگ Deep Linking (`catalogs.length === 0`) → **رفع شده**؛ `useEffect` حالا به `[catalogs]` وابسته است و با شرط `catalogs.length > 0` اجرا می‌شود (`App.tsx:77-96`).
- ضعف `dateToNumber` → **بهبود یافته**؛ regex انعطاف‌پذیر `\D+` حالا فرمت‌های `۱۴۰۴/۱۰/۹` و `۱۴۰۴-۱۰-۰۹` را پوشش می‌دهد (`App.tsx:30-34`).

**نتیجه:** اگر پلن قبلی را مو‌به‌مو اجرا کنید، هفته‌ها روی کارهای انجام‌شده وقت می‌گذارید و دو مشکل واقعاً خطرناک باقی می‌مانند.

## ۱. 🔴 مشکلات واقعاً بحرانی (که پلن قبلی ندید)

### ۱.۱ توکن بات بله به‌صورت plaintext داخل گیت commit شده — نشت راز
- در `public/submit_form.php:98-99` توکن واقعی بات و `chat_id` **هاردکد و داخل ریپازیتوری** است:
  ```php
  $token = "1804437858:DetvFi-...";  // در تاریخچه گیت برای همیشه ثبت شده
  $chat_id = "1910025184";
  ```
- **خطر:** هر کسی با دسترسی به ریپو (یا اگر روزی public شود) می‌تواند از بات شما پیام ارسال کند یا آن را مسموم کند. حتی اگر فایل را الان پاک کنید، توکن در history گیت می‌ماند.
- **راه‌حل فوری:**
  1. توکن را در پنل BotFather بله **revoke / regenerate** کنید (مهم‌ترین قدم — فرض کنید لو رفته).
  2. توکن جدید را به یک فایل خارج از ریپو منتقل کنید: `config.php` (در `.gitignore`) یا متغیر محیطی سرور (`getenv('BALE_TOKEN')`).
  3. `submit_form.php` و `requests.php` و `rate_*.txt` را به `.gitignore` اضافه کنید.

### ۱.۲ رمز پنل ادمین وبسایت — plaintext داخل باندل
- در `components/AdminLogin.tsx:16` رمز به‌صورت literal در سورس است و در باندل production نمایش داده می‌شود:
  ```ts
  if (password === 'JU=:zzBTj.Cg4m*...') { onLogin(); }
  ```
- این یک auth کاملاً client-side است؛ حتی بدون رمز هم می‌توان با باز کردن React DevTools و ست‌کردن state، `isAuthenticated=true` کرد.
- **اما توجه:** پنل ادمین فقط داده را در **localStorage مرورگر همان شخص** تغییر می‌دهد (هیچ نوشتنی روی سرور نیست). یعنی این «نفوذ» در عمل به کسی اجازه تغییر سایت بقیه را نمی‌دهد. پس این مشکل **متوسط** است، نه فاجعه‌بار — برخلاف لحن پلن قبلی.
- **راه‌حل:** اگر پنل قرار است واقعاً داده را روی سرور بنویسد (بخش ۲.۲)، آن‌وقت auth باید سمت سرور برود. تا آن زمان حداقل رمز را از env در build بخوانید (`import.meta.env`) تا در سورس کنترل نباشد.

### ۱.۳ چت‌بات وبسایت اصلاً هوش مصنوعی ندارد — فقط placeholder
- `public/chat.php:47` یک پاسخ ثابت برمی‌گرداند؛ هیچ اتصالی به Gemini/AI نیست.
- پکیج `@google/genai` در `package.json` هست اما **در هیچ فایلی استفاده نشده** (dependency مرده).
- یعنی روی **وبسایت**، دکمه «سوال در مورد محصول» در عمل پاسخ هوشمند نمی‌دهد — برخلاف ادعای README. هوش مصنوعی واقعی فقط در **افزونه وردپرس** وجود دارد.
- **تصمیم معماری لازم (سوال کلیدی):** آیا چت‌بات وبسایت باید مستقل بماند یا به همان موتور افزونه وردپرس وصل شود؟ این تصمیم کل فازبندی AI وبسایت را تعیین می‌کند (بخش ۶).

### ۱.۴ فایل‌های PHP بدون authentication و CORS باز
- `chat.php` و `submit_form.php` با `Access-Control-Allow-Origin: *` و بدون توکن در دسترس عموم‌اند.
- rate limiting با فایل `rate_$ip.txt` و `$_SERVER['REMOTE_ADDR']` قابل دور زدن است (و با هر IP یک فایل جدید می‌سازد → مشکل تعداد inode).
- **راه‌حل:** یک توکن مشترک ساده (هدر `X-Form-Token`) + محدود کردن `Allow-Origin` به دامنه‌ی واقعی + جابه‌جایی rate limit به یک فایل/دیتابیس متمرکز.

## ۲. وضعیت واقعی وبسایت (React SPA) — باگ‌ها و کمبودها

### 🟠 مهم
- **۲.۱ بدون Error Boundary:** در `App.tsx` هیچ `<ErrorBoundary>` دور `Suspense` نیست؛ کرش `BookViewer`/`VideoPlayer` → صفحه سفید. (تأیید شد، هنوز باز است.)
- **۲.۲ معماری داده شکننده:** داده از `window.NAFAS_DATA` (فایل `public/data.js`) می‌آید و تغییرات ادمین فقط در localStorage ذخیره می‌شود (`CatalogContext.tsx:84-90`). هیچ persist سمت سرور نیست → برای انتشار محتوا باید دستی `data.js` را ویرایش کرد. این **هسته‌ی مشکل** است و بیشتر امکانات آینده به آن وابسته‌اند.
- **۲.۳ جستجوی محدود:** فقط `title` و `description` (`App.tsx:156-162`). فیلد `toc` در `types.ts` تعریف شده ولی در جستجو استفاده نمی‌شود.

### 🟡 UX
- **۲.۴ دکمه تغییر تم روی موبایل هست** (در هدر، `App.tsx:238`) — برخلاف ادعای پلن قبلی. اما جستجوی موبایل جدا زیر هدر است (`App.tsx:248`) که می‌تواند یکپارچه شود.
- **۲.۵ بدون صفحه ۴۰۴/خطا:** اگر `cat` در URL پیدا نشود، silently به خانه برمی‌گردد بدون پیام.
- **۲.۶ یادداشت‌های BookViewer فقط localStorage:** بدون export/backup (تأیید شد طبق README).
- **۲.۷ PWA روی `autoUpdate` است** (`vite.config.ts:12`) و `data.js` در `includeAssets` precache می‌شود — یعنی آپدیت محتوا نیازمند آپدیت Service Worker است و کاربر «بی‌صدا» در رفرش بعدی می‌گیرد؛ مکانیزم «نسخه جدید در دسترس است» وجود ندارد.

## ۳. وضعیت واقعی افزونه وردپرس (نسخه ۳.۱.۰) — تقریباً بالغ

افزونه بسیار جلوتر از چیزی است که پلن قبلی فرض کرده بود. ساختار: ۵ جدول دیتابیس (submissions, chatlog, qa, kb, stats)، نسخه DB ۶ با migration خودکار، پشتیبانی چند provider (Gemini/OpenAI/Claude/Custom).

**آنچه قبلاً کامل است:** nonce روی همه endpointها، رمزگذاری AES-256 کلیدها، capability check، timeout، BOM در CSV، lazy-load ویجت/شورتکد، QA bank با fuzzy match، **RAG واقعی** (KB chunking + FULLTEXT + hybrid retrieval)، داشبورد آمار + CSAT، محدودیت آپلود ۲MB.

**آنچه واقعاً باقی مانده (کمبودهای واقعی افزونه):**
- **۳.۱ بدون retry / status-tracking برای بله و تلگرام:** ارسال fire-and-forget است (`class-nafas-chatbot-ajax.php:1330-1352`)؛ شکست بی‌صدا گم می‌شود. → ستون وضعیت ارسال + صف retry.
- **۳.۲ بدون bulk actions در جدول درخواست‌ها:** فقط عملیات تک‌ردیفی. → چک‌باکس + حذف/تغییر وضعیت دسته‌ای.
- **۳.۳ بدون streaming پاسخ AI:** پاسخ کامل JSON برمی‌گردد. → SSE برای نمایش کلمه‌به‌کلمه (بهبود UX، نه بحرانی).
- **۳.۴ بدون webhook ورودی** برای اتصال به CRM/سیستم بیمارستانی.

## ۶. تصمیم معماری کلیدی: چت‌بات وبسایت

سه گزینه برای رفع placeholder بودن `chat.php`:

1. **اتصال وبسایت به موتور افزونه وردپرس** (پیشنهادی اگر وردپرس روی همان دامنه/زیردامنه است): یک endpoint REST در افزونه که SPA آن را صدا بزند → از RAG، QA bank، آمار و امنیت موجود افزونه دوباره استفاده می‌شود. کمترین کار، بیشترین ارزش.
2. **پیاده‌سازی Gemini مستقیم در `chat.php`**: مستقل، اما همه‌ی RAG/امنیت/آمار باید از صفر دوباره ساخته شود (دوباره‌کاری).
3. **حذف چت AI از وبسایت** و نگه‌داشتن فقط فرم‌های ADR/مشاوره: ساده‌ترین، اگر چت AI روی وبسایت اولویت تجاری ندارد.

> این تصمیم باید قبل از فاز یک گرفته شود چون مسیر کل AI وبسایت را تعیین می‌کند.
